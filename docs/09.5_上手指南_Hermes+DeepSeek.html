01

整体流程总览

在动手之前，先看清楚这套工作流里的几个角色，知道每一步在干嘛，后面就不会迷路。

我们的协作大致由这几块拼起来，你会在不同环节用到它们：

Claude Project

团队在 claude.ai 上的项目空间，放着课题资料、报告、查新材料。和 Claude 讨论、写文档、查资料都在这里。

服务器

我们真正跑代码的地方。你用自己电脑上的 VS Code 远程连上去，在上面的「工作区」干活。

GitHub 仓库

xkwd-DC/DC-AIino 是代码的「云端总账本」。你已经加入了 xkwd-DC 组织，所以有权限。

Claude Code

装在服务器终端里的 AI 编程助手。它能读懂整个项目代码，帮你写代码、改 bug、跑命令。

把它串起来就是一条主线：

◷ 一句话流程

先在本机 配好 SSH → 用 VS Code 远程连上服务器 → 在自己的工作区里 clone 仓库 → 在里面写代码并用 Claude Code 协助（用 tmux 让对话不掉线） → 用 Git 把成果 commit & push 回 GitHub。需要查资料、写报告时，就回到 Claude Project。

下面每一节都对应其中一步，照着做就行。完全没接触过命令行也别担心，需要敲的命令我都写好了，点「复制」即可。

02

在 Claude Project 里协作

Project 是我们这个课题的「共享大脑」——资料、上下文和对话都集中在一处，Claude 会基于这些内容回答你。

进入项目

  1. 打开 claude.ai 并用团队账号登录。
  2. 在左侧边栏点 Projects（项目），找到我们这个粮食生产风险可视化的课题项目并点进去。
  3. 项目里有两块：上方/右侧的 项目知识（Project knowledge） 放的是共享资料（论文、研究报告、查新报告、申报材料等）；中间是 对话区，每次新开一个聊天都会自动带上这些资料作为背景。

怎么用才顺手

  • 每个小任务开一个新对话。比如「帮我梳理核心因子识别的方法」和「帮我改研究报告第三章」分开聊，上下文更干净。
  • 需要 Claude 参考的资料就传进项目知识里，而不是每次粘贴一大段。传一次，之后所有对话都能用到。
  • 提问尽量说清背景和目标：你想要什么、给谁看、要多长、什么格式。要求越具体，结果越好。
  • 需要可下载的文件（Word 报告、PPT、Excel）时直接说，Claude 可以生成给你。
✦ 小提示

Project 里的对话只对项目成员可见，是放课题专属上下文的好地方。和别的不相干的事情（比如随便问个常识）就不必占用项目对话，单独开个普通聊天即可。

03

从零配置 SSH

SSH 是安全登录远程服务器的方式。配置一次之后，VS Code 和终端就能「一键」连上服务器、不用每次输密码。如果你的 SSH 配置文件已经由别人帮你配好了，可以直接跳到下一节。

先理解两样东西

  • 一对密钥：在你自己电脑上生成。私钥（没有 .pub 后缀）绝不外传，留在本机；公钥（.pub 结尾）复制到服务器上。服务器认得你的公钥，就放你进门。
  • 一个配置文件：本机的 ~/.ssh/config。在里面给服务器起个「昵称」，VS Code 的连接列表里出现的就是这个昵称，连接也变成 ssh 昵称 这么简单。
⚠ 私钥 = 你的钥匙

私钥文件（如 id_ed25519）等于进服务器的钥匙，千万不要发给任何人、不要上传到 GitHub。需要交换的永远只有公钥（.pub）。

① 生成密钥（本机操作，只需一次）

在你自己电脑的终端里运行（Windows 用 PowerShell 或 Git Bash）：

bash · 在你自己的电脑上
# 邮箱换成你自己的，一路回车即可（位置用默认）
$ ssh-keygen -t ed25519 -C "你的邮箱"
  • 问保存位置时，直接回车用默认（~/.ssh/id_ed25519）。
  • 问 passphrase（口令）时，可以直接回车留空（更方便）；想更安全就设一个——以后连接会让你输这个口令，而不是服务器密码。

② 把公钥送上服务器

最省事的办法（Mac / Linux / Git Bash 都行），这一步会让你输一次服务器密码：

bash · 一键拷贝公钥
# 把 用户名 和 服务器地址 换成实际的
$ ssh-copy-id 用户名@服务器IP

如果系统没有 ssh-copy-id（纯 Windows 常见），就手动：先 cat ~/.ssh/id_ed25519.pub 复制整段公钥，再登录服务器，把它追加到服务器的 ~/.ssh/authorized_keys 文件末尾即可。

③ 写本机的 config 文件

编辑（没有就新建）本机的 ~/.ssh/config，加上这样一段：

~/.ssh/config · 在你自己的电脑上
Host dc-server
    HostName 123.45.67.89
    User 你的服务器用户名
    Port 22
    IdentityFile ~/.ssh/id_ed25519
  • Host：昵称，随便起（这里叫 dc-server）——这就是 VS Code 列表里会显示的名字。
  • HostName：服务器真实 IP 或域名。
  • User：你在服务器上的登录名。
  • Port：端口，默认 22，没改过就写 22（或整行删掉）。
  • IdentityFile：指向你的私钥路径。
◷ Windows 路径

Windows 上这个文件在 C:\Users\你的用户名\.ssh\config。若 .ssh 文件夹不存在，先建一个。

④ 测试连接

bash · 用昵称连接
$ ssh dc-server
# 不再要服务器密码、直接进去 = 配置成功
#（若你给密钥设了口令，这里会问的是口令，不是服务器密码）
⚠ 连不上？十有八九是权限

密钥认证最常见的坑是权限不对，导致 SSH 悄悄拒绝、回退到要密码。在服务器上修正一下即可：chmod 700 ~/.ssh 且 chmod 600 ~/.ssh/authorized_keys。还连不上就加 -v 看详细日志：ssh -v dc-server。

04

VS Code 连接服务器

用本地的 VS Code 远程连上服务器，就像在自己电脑上写代码一样，但文件和运行环境都在服务器上。上一节配好的 SSH 在这里直接派上用场。

第一次：装好插件

  1. 打开 VS Code，点左侧最下方的 扩展（Extensions）图标（四个小方块）。
  2. 搜索 Remote - SSH，安装由 Microsoft 出品的那个（或者直接装 Remote Development 扩展包，里面都包含了）。
  3. 装好后，VS Code 左下角会出现一个蓝绿色的 >< 远程按钮。

连接服务器

  1. 点左下角那个 >< 按钮，在弹出的菜单里选 Connect to Host…（连接到主机）。
  2. 因为 SSH 配置文件（~/.ssh/config）里已经写好了主机，列表里会直接出现你起的昵称（比如 dc-server），点它就行，不用手动敲地址。
  3. 第一次连接可能会让你确认指纹（选 Continue）。如果之前配的是密钥认证，就不用输密码了；如果让你输密码，按提示输入即可。
  4. 连接成功后，左下角会显示 SSH: 服务器名，说明你现在已经「人在服务器上」了。

打开你的工作区

  1. 顶部菜单 File → Open Folder（打开文件夹）。
  2. 这时浏览的是服务器上的目录。进入分配给你的个人工作区文件夹（如果还没有，可以在终端里建一个，见下一节）。
  3. 顶部菜单 Terminal → New Terminal 打开终端，之后所有命令都在这个终端里敲——它直接跑在服务器上。
◷ 两种「SSH」别搞混

① 你的电脑 → 服务器：这个已经配好了，就是上面用的。
② 服务器 → GitHub：这是另一回事，是从服务器拉取/推送代码用的认证，下一节会单独说。

05

在工作区 clone 仓库

「clone（克隆）」就是把 GitHub 上的仓库完整下载一份到服务器上，变成你能编辑的本地文件夹。你已经在 xkwd-DC 组织里，所以有读写权限。

① 先告诉 Git 你是谁（只需一次）

这一步是给你之后的每一次提交署名，全局设置一次即可。在终端里运行：

bash · 在服务器终端
$ git config --global user.name "你的名字"
$ git config --global user.email "你的GitHub邮箱"

② 让服务器能访问 GitHub（认证）

从服务器拉代码需要让 GitHub 认识这台机器。推荐用 SSH 密钥方式，配一次以后都不用输密码：

bash · 生成并查看密钥
# 1) 生成一对密钥（一路回车即可，邮箱换成你的）
$ ssh-keygen -t ed25519 -C "你的GitHub邮箱"

# 2) 打印出公钥，整段复制下来
$ cat ~/.ssh/id_ed25519.pub
  1. 复制上面打印出来的公钥（以 ssh-ed25519 开头的一整行）。
  2. 打开 GitHub → 右上角头像 → Settings → SSH and GPG keys → New SSH key，把公钥粘进去保存。
  3. 验证一下是否连通：
bash · 测试连接
$ ssh -T git@github.com
# 看到 "Hi 你的用户名! You've successfully authenticated" 就成功了
◷ 不想配 SSH？

也可以用 HTTPS + 访问令牌（PAT）：clone 时地址换成 https://github.com/...，遇到要密码时，用 GitHub 个人访问令牌代替密码（在 Settings → Developer settings → Personal access tokens 里生成）。SSH 配一次更省心，按需选择。

③ Clone 仓库到你的工作区

bash · 克隆代码
# 进入（或新建）你的个人工作区目录
$ mkdir -p ~/workspace # 如果还没有就建一个
$ cd ~/workspace

# 用 SSH 方式克隆（推荐）
$ git clone git@github.com:xkwd-DC/DC-AIino.git

# 或者用 HTTPS 方式克隆
$ git clone https://github.com/xkwd-DC/DC-AIino.git

# 进入项目文件夹
$ cd DC-AIino

完成后，在 VS Code 里 File → Open Folder 选中这个 DC-AIino 文件夹，就能开始编辑了。

⚠ 开干之前先拉一下

每次开始工作前，先 git pull 把别人最新的改动同步下来，能减少很多冲突。这个习惯很值钱。

06

Git / GitHub 常用操作

Git 是给代码做「存档」的工具。先理解几个核心概念，命令就很好记了。

先搞懂三个空间

  • 工作区：你正在编辑的文件本身。
  • 暂存区（staged）：你挑出来「准备这次要存档」的改动，用 git add 放进去。
  • 本地仓库：用 git commit 把暂存区的改动正式存成一个「快照」，留在你这台机器上。
  • 远程仓库（GitHub）：用 git push 把本地的 commit 上传到 GitHub，别人才看得到。
✦ 关键词翻译

commit（提交）＝ 在本地给当前改动拍一张存档照片，附一句说明。
push（推送）＝ 把这些存档照片上传到 GitHub。
pull（拉取）＝ 把 GitHub 上别人的最新改动下载到本地。
简单记：commit 是存档，push 是上传，pull 是同步。

日常工作流（最常用的一套）

bash · 一天的典型流程
# 1) 开工前，先同步最新代码
$ git pull

# 2) 改完文件后，看看动了哪些
$ git status

# 3) 把要提交的改动放进暂存区（. 表示全部）
$ git add .

# 4) 提交一个存档，写清楚这次干了啥
$ git commit -m "完成核心因子识别模块的数据预处理"

# 5) 上传到 GitHub，让队友看到
$ git push

用分支（branch）来不互相干扰

多人协作时，最好各自开一条分支干活，做完再合并回主分支，避免直接改乱主线。

bash · 分支操作
# 新建并切换到一条自己的分支
$ git checkout -b feature/data-cleaning

# 在分支上正常 add / commit / push
$ git push -u origin feature/data-cleaning

# 切回主分支
$ git checkout main

推上去后，可以在 GitHub 网页上发起 Pull Request（PR），让队友 review 后再合并进 main。这是最稳妥的协作方式。

常用命令一览

命令作用
git status查看当前有哪些改动、在哪条分支
git add .把所有改动放进暂存区（也可指定单个文件）
git commit -m "说明"把暂存区的改动存为一个本地快照
git push把本地 commit 上传到 GitHub
git pull把 GitHub 上的最新改动同步到本地
git log --oneline简洁地查看提交历史
git diff查看具体改了哪几行
git branch列出所有分支（带 * 的是当前分支）
git checkout -b 名字新建并切换到一条新分支
git clone 地址把远程仓库下载到本地
⚠ 别慌的几个原则

commit 信息写清楚「做了什么」，方便日后回溯；不确定操作前先 git status 看清状态；遇到 push 被拒（别人先推了），先 git pull 再 push。真遇到冲突或搞不定，先停手别乱试，截图发群里问。

07

启动并使用 Claude Code

Claude Code 是跑在终端里的 AI 编程助手。它能读懂整个项目、帮你写和改代码、还能直接运行命令。下面在服务器终端里完成安装和启动。

① 安装

最通用的方式是用 npm 安装（需要服务器上已装好 Node.js 18 及以上）：

bash · 安装 Claude Code
# 先确认 Node 版本（应 ≥ 18）
$ node --version

# 全局安装 Claude Code（不要加 sudo）
$ npm install -g @anthropic-ai/claude-code

# 验证安装成功
$ claude --version
◷ 装不上 / 权限报错？

如果遇到 EACCES 权限错误，不要用 sudo。把 npm 全局目录改到自己有权限的位置即可：
mkdir ~/.npm-global → npm config set prefix '~/.npm-global' → 把 export PATH="$HOME/.npm-global/bin:$PATH" 加到 ~/.bashrc 或 ~/.zshrc，再 source 一下重新安装。Anthropic 也提供「原生安装器」（无需 Node.js），详见 官方文档。

② 启动并登录

  1. 进入项目目录（很重要——它会以当前文件夹为工作上下文）：cd ~/workspace/DC-AIino
  2. 输入 claude 启动。首次运行会引导你登录授权。
  3. 按提示用团队的 Claude 账号完成登录（会给一个链接/验证码的授权流程），授权后凭证保存在本地，之后就不用再登了。
bash · 启动
$ cd ~/workspace/DC-AIino
$ claude

③ 怎么用

启动后进入一个对话式界面，你直接用大白话描述需求就行，比如：

  • 「读一下这个项目，告诉我整体结构和各模块是干嘛的。」
  • 「帮我在 data/ 目录写一个脚本，把气候数据按月份聚合。」
  • 「这段代码报了这个错，帮我定位并修好，然后跑测试验证。」

Claude Code 会自己读相关文件、提出修改、（在你允许后）执行命令。第一次进新项目，建议先让它生成一份项目说明文件：

claude · 斜杠命令
/init      # 自动生成 CLAUDE.md（记录项目命令、结构、约定）
/help      # 查看所有可用命令
/clear     # 清空当前上下文，开始新任务
/compact   # 压缩对话历史，省 token
✦ CLAUDE.md 是个好东西

/init 生成的 CLAUDE.md 会被 Claude 每次开场自动读取，相当于项目的「使用说明」。把它一起 commit 到仓库，全组都能共享同一套上下文，Claude 对项目的理解会更准。

08

tmux：让对话留在服务器

这是最容易被忽略、但极其重要的一步。tmux 能让 Claude Code 在你断开连接后继续留在服务器上运行，下次回来对话还在原地。它好用但有几处「反人类」的设计，下面讲清楚。

为什么必须用它

当你 SSH 连上服务器、直接运行 claude 时，这个进程是「挂」在你这次连接上的。一旦——

  • 笔记本合盖休眠、WiFi 抖动、网络断开；
  • 不小心关掉了 VS Code 或终端窗口；

——连接一断，Claude Code 进程当场被杀掉，正在跑的任务和对话全没了。tmux 的作用就是：在服务器上开一个常驻会话，进程跑在它里面。你「离开（detach）」后会话照常运行，下次「回来（attach）」时，一切原样还在，Claude Code 还停在你离开时的那句对话。

核心三步走

bash · 在服务器终端
# 如果没装（一般已自带）
$ sudo apt install tmux

# 1) 新建一个名叫 claude 的会话
$ tmux new -s claude

# 2) 进去之后，照常干活：进项目、跑 Claude Code
$ cd ~/workspace/DC-AIino
$ claude

# 3) 想离开但让它继续跑 —— 按下「分离」快捷键：
#    先按 Ctrl+B，松手，再单独按 d

分离之后，你可以放心关掉 VS Code、断网、合盖。等下次重新连上服务器，再「接回去」：

bash · 回来接管会话
# 看看现在有哪些会话
$ tmux ls

# 接回名叫 claude 的会话，一切原样还在
$ tmux attach -t claude

# 真正用完了、确认不要了，再杀掉它
$ tmux kill-session -t claude

⚠ 那些「反人类」的地方

tmux 的设计有些反直觉，提前知道能少踩很多坑：

⚠ 1 · 所有快捷键都是「两段式」

tmux 的命令不是同时按。标准动作是：先按住 Ctrl+B（这叫「前缀键」），整个松开，再单独按第二个键。比如分离是「Ctrl+B 松手 d」，不是 Ctrl+B+D 一起按。新手最常卡在这——三个键一起摁，什么都不会发生。

⚠ 2 · 「分离」和「退出」是两码事

分离（detach）：Ctrl+B 然后 d —— 会话继续活着，这才是你想要的。
退出（exit）：在里面输 exit 或按 Ctrl+D —— 会话被关闭、内容丢失。
记牢：要走就「分离」，绝不要 exit。直接关终端窗口反而没事，会话还在。

⚠ 3 · 鼠标滚轮翻不动历史

在 tmux 里直接滚滚轮往往翻不了上面的输出。要看历史得先进「复制模式」：按 Ctrl+B 然后 [，再用方向键 / PageUp 翻，看完按 q 退出。忘了自己在复制模式里乱敲键盘 → 像卡住了一样没反应，这是最日常的困惑。（解法见下方「治本」小贴士。）

⚠ 4 · 会话会偷偷堆积

分离后会话是「看不见」的，时间一长容易攒下一堆忘记关的会话，白白占内存。养成习惯：偶尔 tmux ls 看一眼，用不到的 tmux kill-session -t 名字 清掉。

⚠ 5 · 套娃问题（tmux 里又开了 tmux）

如果不小心在一个 tmux 会话里又进了另一个 tmux，前缀键会被外层抢走。这时要给内层发指令，得把前缀键按两次：Ctrl+B Ctrl+B 再按命令键。最简单的办法是别套娃——进去前先 tmux ls 确认自己在不在会话里。

◷ 底部那条绿条是正常的

进入 tmux 后屏幕最下方会一直有一条绿色状态栏，那是「你正在 tmux 里、受保护」的标志。看不到这条绿条，就说明你不在会话里，工作没被保护——这是判断自己在不在 tmux 的最快方法。

✦ 治本：让滚轮能用

嫌滚动反人类，可以一劳永逸。在服务器上编辑 ~/.tmux.conf，加一行 set -g mouse on，保存后新开会话即可用鼠标滚轮翻历史、点击切换窗格。对新手体验提升巨大。

推荐的每日开场白

把流程固定成一句，每天照敲就行——有会话就接回，没有就新建：

bash · 一行搞定
$ ssh dc-server
$ tmux attach -t claude || tmux new -s claude
$ cd ~/workspace/DC-AIino && claude
# 离开时：Ctrl+B 松手 d
◷ 配合 VS Code

VS Code 的集成终端里同样可以跑 tmux——在远程终端里执行上面的命令即可。VS Code 本身断开会自动重连，但终端里的进程只有放进 tmux 才不会被杀。所以：凡是要长时间跑 Claude Code，先 tmux，再 claude。

安装 Claude Code 技能（Skills）

技能（Skill）是一包包「专项本领」——比如处理 Word/PDF/PPT、做前端设计等。装上之后，Claude Code 在相关任务里会自动调用，特别适合我们这种又要写报告、又要做可视化系统的课题。

从官方仓库安装（推荐）

在 Claude Code 会话里（也就是启动 claude 之后），输入下面的斜杠命令：

claude · 安装官方技能
# 1) 把 Anthropic 官方技能仓库注册为「插件市场」
/plugin marketplace add anthropics/skills

# 2) 安装文档处理技能（Word / PDF / PPT / Excel）
/plugin install document-skills@anthropic-agent-skills

# 3) 安装示例/常用技能合集
/plugin install example-skills@anthropic-agent-skills

# 4) 查看已装好的技能
/skills

装好后，不用做额外配置，在提问里提到相关任务就会自动触发，比如：「用 PDF 技能把这份查新报告里的表格提取出来」「用 docx 技能帮我把研究报告排成正式的 Word 文档」。

从任意 GitHub 仓库手动安装

如果想装某个 GitHub 上的开源技能，可以直接把它放进技能目录：

bash · 手动安装
# 个人级（对所有项目生效）：克隆进 ~/.claude/skills/
$ git clone https://github.com/作者/技能名.git ~/.claude/skills/技能名

# 项目级（只对当前仓库生效）：放进项目里的 .claude/skills/
$ git clone https://github.com/作者/技能名.git .claude/skills/技能名
⚠ 三个常见坑

① SKILL.md 必须在技能文件夹的第一层（路径形如 ~/.claude/skills/技能名/SKILL.md），不能再嵌一层。
② 安装非官方技能前，先大致看一眼它的内容，确认来源可信再用。
③ 装完用 /skills 确认是否加载成功；没出现就检查路径和 SKILL.md 的格式。

✦ 我们项目适合先装哪些

报告/查新材料多 → document-skills（Word/PDF/PPT/Excel）很实用；要做「可视化系统」的界面 → 可以找 frontend-design 相关技能。先把这两类装上，基本够用了。

使用 Hermes Agent 框架 · 接入 DeepSeek

Hermes Agent 是我们本地部署的 AI 智能体框架，跑在服务器上，支持接入各种大模型。它和 Claude Code 是同类工具但定位不同——Hermes 更偏向「全功能自主智能体」，能跨平台（终端、QQ、Telegram）、有持久记忆、可以调度子代理、还能安排定时任务。

我们已经在服务器上装好了 Hermes Agent（v0.15.1），并配置了 DeepSeek Chat 作为主模型。你在服务器上可以随时用，按需选择用 Claude Code 还是 Hermes。

Hermes vs. Claude Code：怎么选

两个都是「AI 编程助手」，但各有侧重点。了解它们各自擅长什么，方便你按场景选工具：

方面Hermes AgentClaude Code
底层模型DeepSeek Chat（目前），可随时切换Claude（Anthropic）
代码能力好用，适合中大型代码任务顶尖，行业标杆
记忆✅ 跨会话持久记忆，记住你的偏好❌ 每次新会话从头开始
技能系统✅ 自动保存工作流为 skills✅ 也有 /skills 体系
多平台✅ QQ、Telegram、Discord 等都能用❌ 仅终端
子代理调度✅ delegate_task 派并行子代理❌ 单线程
定时任务✅ 内置 cron 调度❌ 无
MCP 服务器✅ 原生支持✅ 也支持
开/闭源✅ 完全开源（Nous Research）⚠ Anthropic 专有
费用DeepSeek API 极便宜，走自己的 key需 Claude 订阅或 API key
✦ 典型用法推荐

写代码主力用 Claude Code（代码能力最强），需要记忆、跨平台推送、定时跑任务、并行子代理时切到 Hermes。两个互补，不冲突。

① 快速启动 Hermes

bash · 在终端启动 Hermes
# 直接启动（进入交互式对话）
$ hermes

# 带项目上下文启动（建议在项目目录里）
$ cd ~/workspace/DC-AIino
$ hermes

# 查看当前模型和 Provider
$ hermes config get model.default
$ hermes config get model.provider

启动后你会在一个交互式终端里和 Hermes 对话。它的工具比 Claude Code 更丰富——可以读文件、写代码、跑 shell、搜网络、看图片等。

② 单次查询（非交互）

不想进对话界面，只想让 Hermes 做一件事就走，用 -q 模式：

bash · 一次查询
# 单次提问
$ hermes chat -q "这个项目有哪些 Python 文件？帮我梳理一下结构"

# 让 Hermes 在后台执行一个任务
$ hermes chat -q "把 data/ 目录下的 CSV 按月份聚合，输出到 output/agg.csv"
◷ 同样建议先开 tmux 再跑

和 Claude Code 一样，长时间运行的 Hermes 任务也建议放到 tmux 会话里跑，防止断网中断。用法完全一样——tmux new -s hermes 再启动 hermes。

③ 配置 DeepSeek 模型

服务器上已经配好的默认就是 DeepSeek Chat。如果你想自己从头配一个新实例，步骤很简单：

  1. 注册 DeepSeek 开发者平台，登录后在 API Keys 页面生成一个 key。
  2. 充值（DeepSeek 极其便宜，首次注册通常送几十万 token，之后约 ¥1/百万 token，一个项目跑下来也就几毛钱）。
  3. 把 API key 填入服务器上的环境变量：
bash · 在服务器上配置 DeepSeek key
# 编辑 .env 文件
$ hermes config env-path
# 会输出 ~/.hermes/.env
$ nano ~/.hermes/.env
# 添加（或修改）下面两行：
DEEPSEEK_API_KEY=sk-你的key
DEEPSEEK_BASE_URL=https://api.deepseek.com

# 设置默认 model
$ hermes config set model.default deepseek-chat
$ hermes config set model.provider deepseek

# 验证配置
$ hermes doctor
$ hermes chat -q "你好，请用一句话介绍自己"
◷ DeepSeek 的优势

① 价格低——约为 Claude 的 1/20~1/50；② 中文能力强——原生支持中文，在代码和中文混合场景下表现稳定；③ 上下文窗口大（1M tokens，可以一次性读入大量代码文件）；④ 支持 FIM（Fill-In-the-Middle） 代码补全，适合 IDE 集成。缺点是综合推理和复杂代码生成仍不及 Claude Opus。

④ 切换模型 / Provider

Hermes 最大的优势之一是模型和 Provider 解耦——你可以在不改变任何工作流的前提下，一键切换不同的模型：

bash · 交互式切换模型
# 交互式选择（推荐——会列出所有可用模型）
$ hermes model

# 或直接指定 Provider + 模型
$ hermes config set model.default deepseek-chat
$ hermes config set model.provider deepseek

# 如果想用 OpenRouter 的多模型路由
$ hermes config set model.provider openrouter
$ hermes config set model.default "anthropic/claude-sonnet-4"

# 临时指定模型（不修改配置）
$ hermes chat -m "openai/gpt-4o" -q "帮我写一个快速排序"

支持的 Provider 包括：DeepSeek、OpenRouter（可路由到 200+ 模型）、Anthropic、OpenAI、Google Gemini、xAI/Grok、本地 Ollama、HuggingFace 等 20+。

⑤ Hermes 的独有功能——持久记忆

Claude Code 每次新开会话都是"从零开始"，但 Hermes 有跨会话的持久记忆。你告诉它的事情它会记住，下次打开还能记起：

hermes · 会话内
# 告诉 Hermes 记住你的偏好
"记住我喜欢用 pytest 运行测试"
"项目里的 climate 数据在 data/raw/climate/ 目录下"

# 下次新会话，Hermes 会自动载入这些记忆
# 不需要重复说明

它还自带技能（Skills）系统——当 Hermes 完成一个复杂任务并踩过坑之后，可以把整个过程存为一个 Skill，下次同类任务直接复用。

⑥ Hermes Skills 速查

Hermes 自带了大量实用技能，可以在会话中随时加载：

hermes · 技能操作
# 列出所有可用技能
$ hermes skills list

# 浏览技能市场
$ hermes skills browse

# 在会话中加载一个技能
/skill github-pr-workflow

# 搜索相关技能
$ hermes skills search "document"

⚠ 两个易混淆的点

① Hermes 的 Skills 和 Claude Code 的 Skills 不是同一个东西——它们各自有一套技能体系，不能混用。如果你在 Hermes 里装了 skill，Claude Code 是看不到的，反之亦然。
② Hermes 不是 Claude Code 的替代品——它们是互补工具。纯代码场景 Claude Code 更强；需要持久记忆、跨平台推送、子代理并行、定时任务时用 Hermes。
