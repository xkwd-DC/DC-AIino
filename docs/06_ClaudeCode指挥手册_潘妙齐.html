PART · 00

核心理念

Claude Code 不是 IDE，是"会编程的实习生"

你的角色变了——以前你是写代码的，现在你是指挥别人写代码的。这意味着：

  • ✓ 你的核心技能变成"会拆任务、会写需求、会审代码、会兜底"
  • ✓ 不要事事都让 AI 写，你心里要有总体架构图
  • ✓ 每个 prompt 之前问自己：我清楚我要它做什么吗？
  • ✗ 不要让它一次性建全站，分阶段、分模块、可验证

一条铁律：每个改动必须可回滚

Claude Code 偶尔会做出灾难性改动（删错文件、改坏依赖）。所以：

bash · 分支管理铁律
# 每个阶段开始前
git checkout -b feature/phase-X
git add -A && git commit -m "checkpoint before phase X"

# 阶段结束确认 OK
git checkout main
git merge feature/phase-X

# 改坏了就丢
git checkout main
git branch -D feature/phase-X
⚠ 不容妥协
所有 Claude Code 的改动都在分支上做，主分支只接受验证过的代码。

Prompt 2.2：把推理接到 API

Prompt · 推理 API
现在把推理服务接到 Flask API。

新建 backend/api/predict.py，包含三个路由：

1. POST /api/predict/xgboost
   body: {features: {Temp:14.0, SPEI:-0.08, ...所有10个特征}}
   response: {risk: 0.0249, shap_values: {Temp: 0.0015, ...}}

2. POST /api/predict/lstm
   body: 同上
   response: {risk: 0.0241}

3. POST /api/predict/dual  # 双模型协同验证
   body: 同上
   response: {
     xgboost_risk: 0.0249,
     lstm_risk: 0.0241,
     consensus: 0.0245,
     divergence: 0.0008,
     shap_values: {...}
   }

要求：
1. 用 pydantic 校验请求 body 的格式
2. 异常处理：缺失特征返回 400 + 具体哪个特征缺
3. 推理时间记录在 X-Inference-Time 响应头里
4. 用 LRU cache 缓存相同输入的推理结果（maxsize=128）

写完后启动服务，用 curl 测一下：
curl -X POST http://localhost:5000/api/predict/dual \
  -H "Content-Type: application/json" \
  -d '{"features":{"Temp":14.2,"SPEI":-0.2,"Flood_R":4.2,...}}'
