## §1 设计动机:为什么不是 hard-code

国家级评审极可能针对申报书 v6 §5.2 #4 的"韧性路径 11 规则引擎"提出质疑:**"这一组 if-else 规则跟 AI 没关系,本质就是 hard-code 政策文本,凭什么算项目核心交付?"** 本节先正面回答这个问题。

本规则引擎采用 **"白箱规则触发条件 + 双视角模型加权"** 架构,与端到端神经网络生成路径相比有三个不可替代的优势:① **可审计** — 每条规则的触发条件 / 推荐动作 / 适用省份均以源代码形式显式可读(`recommendation.ts` 全文 339 行,所有阈值直接出现在 `trigger`/`weight` 闭包里);② **可解释** — 推荐结果可逐项追溯到"哪条规则被哪个省份的哪个数值触发 + 在 weight 排序中位列第几",归因链 100% 闭合;③ **可调参 / 可重训驱动** — 规则触发的阈值(如 `flood > 3.5`、`spei < -0.5`、`irr < 70`)对应 XGB-SHAP 在 v3 数据上识别出的"省内边际贡献分位拐点",Att-LSTM 11 维 softmax attention 权重决定 weight 排序里"稀疏致灾信号"是否压制"长期结构问题";v4 数据落地或模型重训后,SHAP 分布与 attention softmax 会自动迁移,本规则引擎里参数化的阈值与权重表也随之更新——**模型是参数源,规则是政策语言映射层**,二者不是替代关系而是合成关系。这套架构刚好回应了 §3.4(决策系统作为核心交付)的定位:模型空间外推上限(leave-province-out 中位 R² = -16.83)不影响本规则引擎在 31 个已见省份上的可用性。

## §3 SHAP × Attention 双视角合成逻辑

代码层面,合成发生在 `buildPathway(p: ProvinceProfile)` 函数(`recommendation.ts:312-332`)的两个阶段:① **触发阶段** — 对 12 条规则逐个调用 `rule.trigger(p)`,通过 `Array.filter` 收集所有"trigger 返回 true"的规则;② **加权排序阶段** — 对每条触发的规则计算 `rule.weight(p)`,然后 `Array.sort((a, b) => b.w - a.w)` 按 weight 降序排列;③ **阶段切片** — 按 short / mid / long 三个时间桶分别 `slice(0, 4)` 各取前 4 条动作作为最终路径。

**双视角融合体现在 weight 函数的设计上**:① 规则 1—6 的 weight 是 SHAP 视角的体现 — `p.flood` / `|p.spei|` / `80 - p.irr` / `p.temp - 17` / `(1700 - p.sun) / 100` / `7 - p.temp` 这些表达式直接对应 XGB-SHAP 在 v3 数据上揭示的"该因子偏离 baseline 越大、单产边际损失越严重"的统计规律(`rank_corr_v3_card.md` 中 XGB-SHAP Top-3 = NDVI / Prec / Temp 的尺度感);② 规则 7—10 的 weight 是 Attention 视角的体现 — 固定数值(2.0 / 1.5 / 1.2 / 1.0)对应 Att-LSTM softmax attention 对"稀疏致灾信号"(台风 / 寒地霜冻 / 主产区结构压力 / 城郊水资源约束)给予的结构性权重(`rank_corr_v3_card.md` 中 Att-LSTM Top-3 = Drou_A / SPEI / Temp 的"主动关注"逻辑);③ **冲突合成机制** — 当一个省份同时被多类规则触发时(如河南既是 `major` 主产区 weight=1.0 又满足 `flood > 3.5` weight=4.2),Array.sort 让 SHAP 边际贡献大的硬阈值规则在前、Attention 结构性规则在后,**SHAP × Attention Spearman ρ = -0.055 的互补性**(申报书 v6 §3.2)在此具象化为"两套视角并列进入排序,各自抢前 4 名"而非"互相否决"。这种合成不强求两类方法 Top-3 排名一致(因为实证 ρ ≈ 0),也不让某一类视角独占输出,**与 v6 "互补而非一致"叙事完全对齐**。

## §4 三阶段时间线

**短期 short(0-1 年,sub-tag = 监测 / 预警 / 培训)**:聚焦"立即可启动、低成本、不需新建大型工程"的动作,模板包括气象-水文-遥感监测网络部署、墒情自动监测站、SPEI 滚动评估、穗期高温预警等(规则 1-7 / 9-10 均提供 short 动作,规则 8 cold_belt 也覆盖)。

**中期 mid(1-3 年,sub-tag = 基础设施 / 品种结构 / 技术推广)**:聚焦"需要 1-3 年财政 + 农业部门联动的基建 + 品种 + 技术升级",模板包括排涝标准提升至 10 年一遇 / 灌溉率提升至 75% / 耐旱耐热耐冷品种推广 / 农机社会化服务等(全部 12 条规则均提供 mid 动作)。

**长期 long(3-5 年,sub-tag = 制度建设 / 产业升级 / 协同治理)**:聚焦"跨部门跨省协同 + 制度建设 + 战略储备",模板包括跨省流域韧性治理 / 水权交易制度 / 气候适应性育种联合体 / 都市群协同应急储备等(规则 5 low_sun 与 规则 6 cold 不提供 long 动作,其余 10 条规则均覆盖)。

**注**:申报书 v6 §3.4 与论文 outline §6.2 提到"近期 1-2 年 / 中期 3-5 年 / 远期 5-10 年"是政策语言层的标称口径;代码 `ResiliencePath.vue` 实际渲染为 "0-1 年 / 1-3 年 / 3-5 年",sub-tag 全部沿用前端展示文案。两组口径在叙事层等价(政策语言粒度更粗),以代码为渲染权威。

## §5 可调参清单

| 参数 | 默认值 | 单位 | 出现位置(`recommendation.ts`) | 调整影响 |
|---|---|---|---|---|
| 洪涝触发阈值 | `3.5` | % | rule 1 `flood.trigger` | 调高 → 仅最高洪涝省触发;调低 → 更多省进入排涝建设管线 |
| 灌溉率触发阈值 | `70` | % | rule 2 `irrigation.trigger` | 调高 → 更多省进入节水灌溉升级名单(默认 31 省里约 23 省命中) |
| 灌溉 weight baseline | `80` | (常数) | rule 2 `irrigation.weight` | 影响 weight 排序中灌溉规则与 flood/drought 的相对位次 |
| SPEI 干旱触发阈值 | `-0.5` | (SPEI 单位) | rule 3 `drought.trigger` | 调更负 → 仅西北深度干旱省触发;调更宽 → 中部偏干省也进入抗旱预案 |
| 高温触发阈值 | `18` | °C | rule 4 `heat.trigger` | 调高 → 仅华南热带省触发;调低 → 长江中下游也进入耐热品种推广 |
| 日照触发阈值 | `1700` | h / 年 | rule 5 `low_sun.trigger` | 影响西南阴雨区(贵州 / 重庆 / 四川)的触发覆盖 |
| 低温触发阈值 | `6` | °C | rule 6 `cold.trigger` | 影响东北寒地 + 青藏高寒省的覆盖范围 |
| 台风固定 weight | `2.0` | (排序权重) | rule 7 `typhoon.weight` | 调高 → 沿海省份台风规则在最终路径中始终排前列 |
| 寒地固定 weight | `1.5` | (排序权重) | rule 8 `cold_belt.weight` | 调整黑土区在最终路径中的相对位次 |
| 主产区固定 weight | `1.0` | (排序权重) | rule 9 `major.weight` | 主产区结构性建议在最终路径中的优先级 |
| 城郊固定 weight | `1.2` | (排序权重) | rule 10 `urban.weight` | 都市农业在最终路径中的优先级 |
| 智慧农业默认 weight | `0.7` | (排序权重) | rule 11 `tech.weight` | 兜底数字化建议排位 |
| 储备布局默认 weight | `0.5` | (排序权重) | rule 12 `reserve.weight` | 兜底战略储备排位 |
| 每阶段最大动作数 | `4` | 项 | `MAX_PER_STAGE` 常量 | 调高 → 路径更长更密;调低 → 仅保留最高 weight 的前 N 项 |
| 预算量级阈值 | 总动作数 `>= 10` → 高 / `>= 7` → 中 / 其余 → 低 | (无单位) | `estimateBudget` 函数 | 显示给政策决策者的预算量级提示 |

**模型重训驱动的更新点**:`flood / irrigation / drought / heat / low_sun / cold` 六组阈值与 weight 表达式,需要在 paper_panel_v4 落地、XGB-SHAP / Att-LSTM 重训后,依据新的 SHAP 边际分位与 attention softmax 分布更新一次。这是"模型 → 规则"的自动迁移路径,不是 hard-code。
